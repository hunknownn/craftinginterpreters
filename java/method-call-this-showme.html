① 문제 — 메서드는 한 벌, 인스턴스는 여럿

class Greeter { greet() { print this.name; } // 메서드는 클래스에 딱 1개 } var a = Greeter(); a.name = "A"; var b = Greeter(); b.name = "B"; a.greet(); // A 가 나와야 함 b.greet(); // B 가 나와야 함

같은 greet 코드 한 벌인데 this.name이 호출에 따라 A도 되고 B도 되어야 한다. 메서드 본문 안엔 a·b 같은 변수가 안 보이는데, 어떻게? → 호출 직전에 this를 담은 환경을 그때그때 끼워넣는다(bind).

핵심

a.greet를 읽는 순간 get()이 공유 메서드를 그 인스턴스에 bind한 새 LoxFunction으로 만들어 돌려준다. bind는 "this=리시버"가 든 환경[2]를 본문[3]과 closure[1] 사이에 끼워넣을 뿐. 그래서 같은 코드라도 a로 호출하면 this=a, b로 호출하면 this=b — this 조회는 그냥 환경 체인을 한 칸 올라가 변수 찾기다.

③ 나란히 비교 — 딱 한 칸만 다르다

a.greet() 의 체인

[3] 본문
↓
[2] bind · this = a
↓
[1] closure (공유)

b.greet() 의 체인

[3] 본문
↓
[2] bind · this = b
↓
[1] closure (공유)

[1] closure와 [3] 본문 구조는 동일하다. 유일한 차이는 [2] bind 환경의 this 값. 메서드 객체조차 원본은 하나고, bind가 매 호출마다 this만 다른 복사본을 찍어낸다.

⑤ 다른 언어와 같은 패턴

Python
a.greet는 bound method 생성 — self가 미리 묶인 함수. 정확히 같은 발상.
JavaScript
fn.bind(obj) → this 고정된 새 함수. 이름까지 동일.
핵심 일반화
"메서드 = 리시버가 묶인 함수". Lox는 이를 환경에 변수 하나 끼우기로 구현.
