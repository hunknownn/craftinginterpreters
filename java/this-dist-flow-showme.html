📏 distance로 잇는 this — 계산(컴파일타임) → 사용(런타임)

Resolver가 "this까지 몇 칸"을 미리 세고(dist), Interpreter가 그 숫자로 환경 체인을 정확히 거슬러 올라간다

class Greeter { greet() { print this.name; } } var a = Greeter(); a.name = "A"; a.greet();

왜 두 스택이 같은 모양인가

Resolver의 스코프 스택과 Interpreter의 환경 체인은 똑같은 중첩 구조로 만들어진다(class scope ↔ bind env, method scope ↔ 본문 env). 그래서 컴파일타임에 센 dist=1이 런타임에서 그대로 hop 수가 된다 — 실행할 때 이름으로 헤매며 찾지 않고 getAt(1,"this")로 한 번에 정확히 도달한다.

코드 매핑

단계코드하는 일
생성Resolver.visitClassStmtbeginScope 후 scopes.peek().put("this", true) — this 자리 마련
계산Resolver.visitThisExpr → resolveLocal안쪽 스코프부터 "this" 탐색 → distance 계산 → interpreter.resolve(expr, dist)
저장locals: Map<Expr,Integer>이 This 노드 → dist 기록
사용Interpreter.visitThisExpr → lookUpVariablelocals.get(expr)로 dist 꺼내 environment.getAt(dist,"this")

참고: bind는 런타임에 this 값을 환경에 넣는 쪽(쓰기). Resolver는 그 환경이 몇 칸 위일지를 미리 아는 쪽(주소 계산).
