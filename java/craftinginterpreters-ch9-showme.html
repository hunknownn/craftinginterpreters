if (temperature > 30) { print "ice coffee"; } else { print "hot coffee"; }

2. and/or: 값을 보존하는 단축 평가

오른쪽을 평가할까?

Lox의 논리 연산자는 항상 Boolean을 반환하지 않습니다. 왼쪽 값만으로 결과가 정해지면 그 값을 그대로 반환합니다.

left 평가

대기 중

?
right 평가

필요할 때만 평가

논리 로그

선행 개념

isTruthy(): nil과 false만 falsey다.

해결 메커니즘

or는 왼쪽이 truthy면 즉시 반환, and는 왼쪽이 falsey면 즉시 반환한다.

트레이드오프

불필요한 계산과 부작용을 피하지만, 오른쪽이 실행되지 않을 수 있다는 점을 알아야 한다.

연결

조건부 기본값, guard clause, lazy evaluation의 기초 패턴이다.

4. for는 새 런타임 기능이 아니라 파서 설탕이다

desugaring 단계

for 전용 인터프리터 방문자는 없습니다. 파서가 initializer, condition, increment를 조합해서 기존 Block과 While AST로 바꿉니다.

for (var i = 0; i < 3; i = i + 1) { print i; }
→
// 변환 전 Stmt body = statement();

변환 로그

등장 원인

while만으로 반복은 가능하지만, 초기화/조건/증가가 흩어져 실수하기 쉽다.

해결 문제

for는 루프 제어 요소를 한 줄에 모아 읽기 쉽게 만든다.

메커니즘

파서가 for를 { initializer; while (condition) { body; increment; } } 꼴로 바꾼다.

대가

문법은 편해지지만, 파서가 AST 조립 책임을 더 많이 가진다.

트레이드오프 요약

루프 횟수와 위험

조건이 계속 truthy면 인터프리터는 계속 body를 실행합니다. 제어 흐름은 강력하지만 종료 조건을 프로그램이 책임져야 합니다.

기능얻는 것주의점
if런타임 조건에 따른 분기실행되지 않는 branch의 효과는 발생하지 않음
and/or필요 없는 평가를 건너뜀항상 Boolean을 반환하지 않음
while조건 기반 반복조건 갱신을 빼먹으면 무한 루프
for흔한 루프 패턴을 압축구현은 파서 desugaring에 의존

// Interpreter: if public Void visitIfStmt(Stmt.If stmt) { if (isTruthy(evaluate(stmt.condition))) { execute(stmt.thenBranch); } else if (stmt.elseBranch != null) { execute(stmt.elseBranch); } return null; } // Interpreter: while public Void visitWhileStmt(Stmt.While stmt) { while (isTruthy(evaluate(stmt.condition))) { execute(stmt.body); } return null; }
// Interpreter: logical public Object visitLogicalExpr(Expr.Logical expr) { Object left = evaluate(expr.left); if (expr.operator.type == TokenType.OR) { if (isTruthy(left)) return left; } else { if (!isTruthy(left)) return left; } return evaluate(expr.right); }
