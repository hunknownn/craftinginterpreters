심화 — this & init

ch12에서 제일 헷갈리는 두 가지를, 환경 체인으로 끝까지 분해한다

한 줄 요약
· this = 메서드를 인스턴스에 bind 할 때 환경에 몰래 심는 변수
· init = 호출 후 항상 this를 반환하도록 특별 취급되는 메서드

PART 1. this 는 어디서 오는가

가장 먼저 풀어야 할 질문: 메서드 본문의 this는 어느 인스턴스를 가리키나? 답은 — 메서드를 꺼내는 순간(get) 그 인스턴스로 고정된다.

문제: 메서드는 클래스가 소유한다

class Cake { taste() { print this.flavor; } // this = 어느 케이크? } var a = Cake(); a.flavor = "choco"; var b = Cake(); b.flavor = "vanilla"; a.taste(); // choco b.taste(); // vanilla ← 같은 메서드인데 this가 다르다

메서드 함수 객체는 클래스에 하나만 있다. 그런데 this는 인스턴스마다 달라야 한다. 어떻게?

해법: bind() — 인스턴스마다 새 환경을 씌운다

// LoxFunction LoxFunction bind(LoxInstance instance) { Environment environment = new Environment(closure); // closure 위에 한 겹 environment.define("this", instance); // this를 박아넣음 return new LoxFunction(declaration, environment, isInitializer); }

instance.get("taste")가 호출되면 → method.bind(this) 가 불린다. 즉 점(.)으로 메서드를 꺼낼 때마다 그 인스턴스의 this가 든 새 LoxFunction이 만들어진다.

핵심: this는 키워드처럼 보이지만, 구현상으로는 그냥 "this"라는 이름의 지역 변수다. bind가 그 변수를 정의하는 환경을 만들어줄 뿐.

bind는 "원본"을 건드리지 않고 "새 객체"를 만든다

closure는 LoxFunction의 인스턴스 필드다. bind는 이 필드를 this 환경으로 채운 별도의 LoxFunction 객체를 새로 만든다. 그래서 같은 메서드인데 객체가 둘이 된다.

원본 메서드 객체 findMethod 결과
declaration = taste()
closure = [클래스 정의 환경]
isInitializer = false

클래스에 하나만 존재 · this 없음

bind→
바인딩된 객체 a.taste 결과
declaration = taste()
closure = [this=a] 환경
isInitializer = false

get 할 때마다 새로 생성 · this 보유

세팅은 bind, 사용은 call. call()의 new Environment(closure)에서 closure는 자기 객체의 필드다. 바인딩된 객체로 call하면 그 필드가 곧 [this=a] → 자연히 this 환경이 부모로 깔린다. a.taste()와 b.taste()는 declaration·closure 부모는 공유하되 this만 다른 객체다.

환경 체인 — a.taste() 실행 순간

④ 메서드 호출 환경 (call)가장 안쪽
(파라미터 없음)
▲ enclosing
③ bind 환경인스턴스별
this = Cake#a 인스턴스
▲ enclosing
② closure (클래스 정의 시점)
Cake = LoxClass
▲ enclosing
① global
a, b = ...

본문에서 this를 찾으면 ④에 없으니 한 칸 올라가 ③에서 Cake#a를 찾는다. b.taste()면 ③의 this만 Cake#b로 바뀐다 — 나머지 체인은 공유.

Resolver는 this 스코프를 미리 깔아준다

// visitClassStmt beginScope(); scopes.peek().put("this", true); // this를 "선언된 변수"로 등록 for (Stmt.Function method : stmt.methods) { FunctionType decl = method.name.lexeme.equals("init") ? INITIALIZER : METHOD; resolveFunction(method, decl); } endScope();

Resolver가 메서드 바깥에 this 스코프를 한 겹 두기 때문에, 본문의 this는 "몇 칸 위로 올라가면 된다"는 거리(depth)가 정확히 계산된다. 런타임의 bind 환경과 깊이가 딱 맞물린다.

정적 검사: 클래스 밖에서 this를 쓰면 currentClass == NONE 이라 "Can't use 'this' outside of a class." 에러.

PART 2. init — 생성자의 특별 규칙

init은 그냥 init이라는 이름의 메서드일 뿐이지만, 3가지 특별 대우를 받는다.

상황일반 메서드init
클래스 호출 시호출 안 됨자동 실행
반환값return 값 or nil항상 this
값을 return하면OK컴파일 에러

① 클래스 호출 → init 자동 실행

// LoxClass.call() LoxInstance instance = new LoxInstance(this); LoxFunction initializer = findMethod("init"); if (initializer != null) { initializer.bind(instance).call(interpreter, arguments); // this=새 인스턴스 } return instance;

그래서 arity()(필요 인자 수)도 init의 파라미터 수를 따라간다.

public int arity() { LoxFunction init = findMethod("init"); return init == null ? 0 : init.arity(); }

② init은 무조건 this를 반환한다

// LoxFunction.call() try { interpreter.executeBlock(declaration.body, environment); } catch (Return returnValue) { if (isInitializer) return closure.getAt(0, "this"); // 빈 return도 this return returnValue.value; } if (isInitializer) return closure.getAt(0, "this"); // 정상 종료도 this return null;
왜 closure.getAt(0, "this") 인가?
init의 closure는 곧 bind가 만든 환경(③)이다. 거기서 거리 0(자기 자신)에 this가 있다. 그래서 return; 만 써도, 끝까지 흘러도 항상 새 인스턴스가 돌아온다.

덕분에 이런 코드가 동작한다:

var foo = Foo(); print foo.init(); // init을 다시 불러도 Foo 인스턴스를 반환

③ init에서 값 반환은 금지 (Resolver)

// visitReturnStmt if (stmt.value != null) { if (currentFunction == INITIALIZER) { Lox.error(stmt.keyword, "Can't return a value from an initializer."); } resolve(stmt.value); }

return;(빈 반환, early-exit용)은 허용. return 값; 만 막는다 — init은 항상 this를 줘야 하므로 다른 값을 주는 건 의미 모순.

전체 흐름: var b = Breakfast("bacon")

Breakfast("bacon") → visitCallExpr → LoxClass.call()
새 LoxInstance 생성
findMethod("init") → init.bind(instance) : this가 든 환경 생성
bind된 init을 call() : 본문에서 this.meat = "bacon" 실행 → 필드 저장
init 끝 → isInitializer라서 this(인스턴스) 반환
LoxClass.call()이 최종적으로 인스턴스 반환 → b에 대입

헷갈리기 쉬운 점: bind 환경은 매번 새로 만들어진다

bind()는 호출될 때마다 new Environment(closure)로 새 환경을 만든다. 그래서 init의 bind 환경과 이후 get의 bind 환경은 서로 다른 객체다.

var b = Breakfast("bacon"); // (1) init.bind(b) → env_A [this=b] b.serve("you"); // (2) serve.bind(b) → env_B [this=b] b.serve("me"); // (3) serve.bind(b) → env_C [this=b]
시점어떤 bindenv 객체this 값
(1) 객체 생성init.bind (call 안)env_Ab
(2) serve 호출serve.bind (get)env_Bb
(3) serve 또 호출serve.bind (get)env_Cb
env 상자는 매번 다르지만(env_A ≠ env_B ≠ env_C), 그 안의 this가 가리키는 인스턴스는 같은 b다.
· 같은 메서드를 두 번 꺼낸 (2)·(3)도 서로 다른 env — get할 때마다 새 bind라서.
· env를 공유할 필요가 없다. this 값(인스턴스)만 같으면 동작은 동일하고, 호출별 파라미터/지역변수는 오히려 매번 새 환경이라야 안전하게 격리된다.
