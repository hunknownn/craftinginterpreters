11장은 문법을 바꾸지 않는다

11장(Resolving and Binding)은 새 구문을 추가하지 않는다. AST를 한 번 더 훑어 각 변수 참조가 몇 단계 바깥 스코프의 어떤 변수인지 미리 계산해 둘 뿐이다. 그래서 문법 형상은 10장(함수)까지와 동일하다. 클래스/상속은 12~13장.

1왜 규칙을 함수로 쪼개는가 — 우선순위의 문제

평평한 규칙 하나로는 1 + 2 * 3의 의미를 정할 수 없다

등장 배경

표현식을 expr → expr op expr 한 줄로만 쓰면 1 + 2 * 3이 (1+2)*3인지 1+(2*3)인지 문법이 결정하지 못한다(모호함). 또 좌재귀라 재귀 하향 파서가 무한 루프에 빠진다.

해결 — 우선순위마다 규칙(=함수) 한 층

연산자 우선순위 단계 수만큼 규칙을 계층으로 쌓는다. 각 규칙은 자기보다 한 단계 더 강하게 묶이는 규칙을 자식으로 호출한다. 낮은 우선순위가 트리의 위쪽(나중에 계산), 높은 우선순위가 아래쪽(먼저 계산)에 온다.

구문 문법 (Statements) — 문장의 큰 구조. 표현식 규칙은 아래 우선순위 사다리에 생성 규칙과 함께 나온다.

// 구문 문법 (Statements) program → declaration* EOF ; declaration → funDecl | varDecl | statement ; funDecl → "fun" function ; function → IDENTIFIER "(" parameters? ")" block ; parameters → IDENTIFIER ( "," IDENTIFIER )* ; varDecl → "var" IDENTIFIER ( "=" expression )? ";" ; statement → exprStmt | forStmt | ifStmt | printStmt | returnStmt | whileStmt | block ; exprStmt → expression ";" ; forStmt → "for" "(" ( varDecl | exprStmt | ";" ) expression? ";" expression? ")" statement ; ifStmt → "if" "(" expression ")" statement ( "else" statement )? ; printStmt → "print" expression ";" ; returnStmt → "return" expression? 표현식 문법 (우선순위 낮음 → 높음) — 위에서 아래로 점점 강하게 묶인다.

")" )* ; arguments → expression ( "," expression )* ; primary → "true" | "false" | "nil" | NUMBER | STRING | IDENTIFIER | "(" expression ")" ;

2실습: 직접 파싱하기

실제 동작하는 재귀 하향 파서(책 11장 문법)가 입력을 토큰화하고 AST로 만듭니다

트리를 읽는 3가지 원칙

원칙의미
① 깊이 = 우선순위강하게 묶이는 연산자일수록 더 아래(깊은 곳)에 놓인다. 깊은 노드가 먼저 계산되므로 곧 우선순위다. *가 +보다 아래.
② 루트 = 마지막 계산트리의 꼭대기 노드가 가장 나중에 실행된다. 1+2*3의 루트가 +인 이유 — 덧셈이 마지막.
③ 기울기 = 결합 방향왼쪽 결합은 while 루프로 파싱돼 왼쪽으로 깊어지고, 할당처럼 오른쪽 결합은 재귀 호출로 오른쪽으로 깊어진다.

파서는 expression → assignment → … → primary 순서로 약한 규칙부터 내려가며 호출한다. 가장 약한 규칙이 먼저 호출돼 트리 위쪽을 차지하고, 가장 강한 primary가 맨 안쪽 잎이 된다.

파서는 토큰을 왼쪽→오른쪽으로 순차적으로 읽는다. 다음 ▶ 으로 한 단계씩 진행하며 ① 토큰 커서 ② 호출 스택 ③ 자라나는 트리를 함께 보고, 파싱 ▶ 끝까지 로 단숨에 최종 AST까지 갈 수 있다.

"완성된 서브트리"가 자라는 원리

재귀 하향 파서는 잎(primary)부터 만들어 위로 합친다(bottom-up). 막 만들어진 노드가 잠시 떠 있다가, 상위 규칙이 이항 노드를 만들 때 자식으로 빨려 들어가 하나로 합쳐진다. 마지막 단계에서 노드가 하나(Program)만 남으면 그게 곧 최종 AST다.

3할당이 까다로운 이유 — l-value

a = 1 과 a + b = 1 의 차이

문제

=의 왼쪽은 "값을 저장할 장소"(l-value)여야 한다. 하지만 파서는 a를 읽는 순간엔 그게 할당 대상인지 그냥 값인지 모른다. 한 글자 앞을 더 봐야(=가 오나?) 알 수 있다.

해결 — 일단 표현식으로 파싱하고, 끝나면 대상으로 "변환"

assignment 규칙은 먼저 한 단계 위(logic_or)를 평범하게 파싱한다. 그 뒤 =가 보이면, 방금 파싱한 결과가 Expr.Variable인지 확인한다. 맞으면 Expr.Assign으로 바꾸고, 아니면 "잘못된 할당 대상" 에러를 낸다. 우변은 다시 assignment를 호출해 a = b = 1 같은 오른쪽 결합을 자연스럽게 처리한다.

실습창에 a + b = 1; 을 넣어 에러를, a = b = 1; 을 넣어 우결합 트리를 확인해 보세요.

4for는 사실 문법에 없다 — 디슈가링

파서가 for를 while 블록으로 "번역"한다

아이디어

for는 while로 표현할 수 있는 편의 문법(syntactic sugar)이다. 인터프리터에 For 노드를 따로 만들지 않고, 파서가 파싱 단계에서 초기화·조건·증감을 분해해 이미 존재하는 Block·While 노드로 조립한다.

for (var i = 0; i < 3; i = i + 1)
  print i;
⇒
{
  var i = 0;           // 초기화 → 바깥 블록
  while (i < 3) {       // 조건 → while
    {
      print i;
      i = i + 1;       // 증감 → 본문 끝
    }
  }
}

확인: 실습창의 for 예제를 파싱하면 AST에 For가 아니라 Block → While → Block 구조가 나오는 걸 볼 수 있습니다. 이게 디슈가링의 증거입니다.
