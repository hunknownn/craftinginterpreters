① 왜 이게 필요한가 — 문제 상황

메서드는 클래스에 딱 하나만 저장됩니다. 그런데 인스턴스는 여러 개죠.

class Point { init(x, y) { this.x = x; } } var a = Point(1, 2); // 같은 init 메서드를 쓰는데 var b = Point(9, 9); // this는 b를 가리켜야 함

→ 똑같은 init 코드인데, 호출할 때마다 this가 다른 인스턴스를 가리켜야 한다. 어떻게? 답: 메서드 본문을 실행하기 직전에, this가 정의된 환경을 그때그때 끼워넣는다.

핵심 메시지

this는 인터프리터의 특별한 기능이 아니라 "this"라는 이름의 평범한 변수입니다. bind()가 하는 일은 단 하나 — 메서드 본문보다 한 단계 바깥에 "this" = 인스턴스가 정의된 중간 환경을 끼워넣는 것뿐입니다. 나머지는 변수 조회와 똑같이 렉시컬 스코프(환경 체인을 위로 거슬러 올라가기)로 해결됩니다.

③ 알아두면 좋은 선행 개념

Environment
변수 이름→값 Map + 부모(enclosing) 포인터. 못 찾으면 부모로 올라간다.
closure
함수가 "태어난 곳"의 환경. LoxFunction이 들고 다닌다.
bind가 새 함수를 반환
원본은 그대로 두고 closure만 바꾼 복제본. 그래서 인스턴스마다 다른 this.
Resolver의 distance
"this가 몇 칸 위에 있는지"를 미리 계산. 실행 시 getAt(distance,"this")로 정확히 도달.

④ 트레이드오프 — 왜 이 방식?

방식얻는 것잃는 것
this를 변수로 (이 구현)인터프리터에 특수 케이스 0개. 클로저·스코프 코드 그대로 재사용. 메서드도 일반 함수처럼 변수처럼 넘길 수 있음.호출마다 환경 1개 추가 생성(약간의 비용). bind된 새 LoxFunction 객체 생성.
this를 특수 필드/레지스터로환경 추가 생성 없음(빠름)인터프리터 곳곳에 this 특수 처리 분기. 중첩 클로저에서 this 캡처가 까다로워짐.

⑤ 다른 언어에서의 같은 패턴

Python obj.method는 "bound method"를 만든다 — 정확히 Lox의 bind()와 같은 발상(self를 미리 묶은 함수 객체).

JavaScript fn.bind(obj)는 this를 고정한 새 함수를 반환한다 — 이름까지 똑같다.

결론 "메서드 = 리시버가 미리 묶인 함수"라는 같은 아이디어를, Lox는 환경 체인에 변수 하나 끼워넣기로 가장 단순하게 구현한 것.
