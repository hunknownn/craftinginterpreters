우리가 만들 것

class Breakfast { init(meat, bread) { // 생성자 this.meat = meat; // 필드 저장 this.bread = bread; } serve(who) { // 메서드 print "Enjoy your " + this.meat + ", " + who; } } var bacon = Breakfast("bacon", "toast"); // 인스턴스 생성 bacon.serve("you");

클래스 호출(Breakfast(...))이 인스턴스를 만들고, .으로 필드·메서드에 접근한다.

새 AST 노드 5개: Stmt.Class, Expr.Get, Expr.Set, Expr.This (+ 기존 Stmt.Function 재사용)

① 클래스 선언 — 파싱

선언문법: class 이름 { 메서드* }

// Parser.java — 모든 선언이 거쳐가는 진입점 private Stmt declaration() { try { if (match(CLASS)) return classDeclaration(); // ← 추가 if (match(FUN)) return function("function"); if (match(VAR)) return varDeclaration(); return statement(); } catch (ParseError e) { synchronize(); return null; } } // class 토큰을 만나면 들어오는 함수 private Stmt classDeclaration() { Token name = consume(IDENTIFIER, "Expect class name."); consume(LEFT_BRACE, "Expect '{' before class body."); List<Stmt.Function> methods = new ArrayList<>(); while (!check(RIGHT_BRACE) && !isAtEnd()) { methods.add(function("method")); // 함수 파서 재사용 } consume(RIGHT_BRACE, "Expect '}' after class body"); return new Stmt.Class(name, methods); }

메서드는 fun 키워드 없는 함수일 뿐 — 기존 function()을 그대로 쓴다.

① 클래스 선언 — 실행

// Interpreter.java — Stmt.Visitor 구현. class 선언문을 실행 @Override public Void visitClassStmt(Stmt.Class stmt) { environment.define(stmt.name.lexeme, null); // ① HashMap<String, LoxFunction> methods = new HashMap<>(); for (Stmt.Function method : stmt.methods) { LoxFunction function = new LoxFunction( method, environment, method.name.lexeme.equals("init")); // isInitializer methods.put(method.name.lexeme, function); } LoxClass clazz = new LoxClass(stmt.name.lexeme, methods); environment.assign(stmt.name, clazz); // ② return null; }
왜 define → assign 2단계? 먼저 이름만 null로 선언해두면, 메서드 본문 안에서 자기 클래스 이름을 참조할 수 있다.

② 인스턴스 생성

클래스를 함수처럼 호출하면 인스턴스가 나온다

// LoxClass.java — 클래스가 곧 Callable. 호출되면 인스턴스를 만든다 class LoxClass implements LoxCallable { @Override public Object call(Interpreter interpreter, List<Object> arguments) { LoxInstance instance = new LoxInstance(this); LoxFunction initializer = findMethod("init"); // 생성자 → 심화 자료 if (initializer != null) { initializer.bind(instance).call(interpreter, arguments); } return instance; } }
// LoxInstance.java class LoxInstance { private LoxClass clazz; private Map<String,Object> fields = new HashMap<>(); }

인스턴스는 클래스 참조 + 필드 맵 이 전부.

Breakfast(...) → 기존 visitCallExpr이 LoxCallable.call()을 부른다. 함수 호출과 같은 길.

③ 프로퍼티 — 파싱

.은 호출 ()과 같은 우선순위로 묶인다

// Parser.java — unary() 다음 단계. 호출·프로퍼티 체인을 묶는다 private Expr call() { Expr expr = primary(); while (true) { if (match(LEFT_PAREN)) { // 함수 호출 expr = finishCall(expr); } else if (match(DOT)) { // ← 프로퍼티 접근 (추가) Token name = consume(IDENTIFIER, "Expect property name after '.'."); expr = new Expr.Get(expr, name); } else break; } return expr; }

a.b.c() → Get(Get(a,b),c) 를 호출 — 왼쪽부터 체인으로 묶인다.

③ 프로퍼티 — 대입(Set)

파서는 =을 만나기 전엔 Get인지 Set인지 모른다

// Parser.java — 좌변을 일단 표현식으로 파싱한 뒤 '='를 보고 재해석 private Expr assignment() { Expr expr = or(); // 좌변: a.b 면 Expr.Get으로 파싱됨 if (match(EQUAL)) { Token equals = previous(); Expr value = assignment(); // 우변 (우결합 재귀) if (expr instanceof Expr.Variable) { Token name = ((Expr.Variable) expr).name; return new Expr.Assign(name, value); } else if (expr instanceof Expr.Get) { // ← 핵심 (추가) Expr.Get get = (Expr.Get) expr; return new Expr.Set(get.object, get.name, value); } error(equals, "Invalid assignment target."); } return expr; }
l-value를 미리 알 수 없으니 먼저 표현식으로 파싱 → 사후에 Set으로 재해석. 변수 대입(Expr.Assign)과 똑같은 전략.

③ 프로퍼티 — 실행

get — 읽기

// LoxInstance.java Object get(Token name) { if (fields.containsKey(name.lexeme)) // ①필드 return fields.get(name.lexeme); LoxFunction method = clazz.findMethod(name.lexeme); // ②메서드 if (method != null) return method.bind(this); throw new RuntimeError(name, "Undefined property."); }

set — 쓰기

// Interpreter.java @Override public Object visitSetExpr(Expr.Set expr) { Object object = evaluate(expr.object); if (!(object instanceof LoxInstance)) throw new RuntimeError(expr.name, "Only instances have fields."); Object value = evaluate(expr.value); ((LoxInstance) object).set(expr.name, value); return value; }

필드는 자유롭게 새로 추가 가능 (선언 불필요).

우선순위: 필드가 메서드보다 먼저 — 같은 이름이면 필드가 메서드를 가린다(shadowing).

④ 메서드 · this · 생성자(init)

여기서부터가 이 챕터의 진짜 핵심 — 환경 체인 이야기

간단히 미리보기:

  • this 는 메서드를 인스턴스에 bind 할 때 생기는 숨은 변수다.
  • Resolver가 메서드 주위에 this 스코프를 하나 더 씌운다.
  • init 은 호출되면 항상 this를 돌려주는 특별한 메서드.
→ 🔍 심화 자료: this & init 완전 분해 bind()가 만드는 환경 체인, INITIALIZER 규칙, return this 까지 — 별도 슬라이드에서 천천히

발표 때는 이 링크로 넘어가 deep-dive 후 다시 돌아옵니다.

정적 검사 (Resolver)

런타임 전에 잡아주는 의미 오류들

this — 클래스 밖 금지

// Resolver.visitThisExpr public Void visitThisExpr(Expr.This expr) { if (currentClass == ClassType.NONE) { Lox.error(expr.keyword, "Can't use 'this' outside of a class."); return null; } resolveLocal(expr, expr.keyword); return null; }

init — 값 반환 금지

// Resolver.visitReturnStmt public Void visitReturnStmt(Stmt.Return stmt) { ... if (stmt.value != null) { if (currentFunction == INITIALIZER) Lox.error(stmt.keyword, "Can't return a value from an initializer."); resolve(stmt.value); } return null; }
Resolver = 컴파일 타임 규칙, Interpreter = 런타임 동작. 역할이 깔끔하게 나뉜다.

정리

새 타입

  • LoxClass — Callable, 메서드 보관 + 인스턴스 팩토리
  • LoxInstance — 클래스 참조 + 필드 맵

새 AST

Stmt.ClassExpr.GetExpr.SetExpr.This

핵심 통찰

  • 클래스 호출 = 인스턴스 생성 (Callable 재사용)
  • Get을 먼저 만들고 = 보면 Set으로 재해석
  • 필드 > 메서드 우선순위
  • this는 bind가 만드는 환경 변수
