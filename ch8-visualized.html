Chapter 8 — Statements and State

표현식만 다루던 인터프리터에 문장(statement)과 상태(state)를 더하다

문장 · 전역 변수 · Environment · 할당 · 스코프

① 문장 ② 전역 변수 ③ Environment ④ 할당 ⑤ 스코프 & 블록 🔍 변수 탐색 데모 🌳 AST 노드

7장까지의 인터프리터는 1 + 2 * 3 같은 표현식 하나를 계산해 값을 돌려줄 뿐이었습니다. 실제 프로그램이 되려면 여러 문장을 차례로 실행하고, 값을 기억(변수)할 수 있어야 합니다. 8장은 이 두 가지 — statements(부수 효과를 내는 실행 단위)와 state(변수에 저장되는 상태) — 를 추가합니다.

핵심 한 줄  표현식은 값을 만들고, 문장은 효과를 낸다. 그리고 변수는 그 효과를 Environment라는 맵에 기억한다.

1문장 (Statements)

프로그램은 이제 "표현식 하나"가 아니라 "문장의 나열"입니다. 두 종류의 문장으로 출발합니다.

표현식 문장 exprStmt

표현식 뒤에 ;. 값보다 부수 효과(예: 함수 호출)를 위해 존재.

brunch("eggs");

print 문 printStmt

Lox에 내장된 문법. 평가 결과를 stdout에 출력.

print 2 + 1; // 3

새 문법 규칙

program→statement* EOF NEW
statement→exprStmt | printStmt NEW
exprStmt→expression ";"
printStmt→"print" expression ";"
인터프리터는 이제 두 개의 Visitor를 구현합니다: Expr.Visitor<Object>(값 반환) 와 Stmt.Visitor<Void>(효과만). 표현식은 evaluate(), 문장은 execute()로 처리.

2전역 변수 (Global Variables)

변수에는 두 동작이 필요합니다 — 선언(이름을 만들고 값을 묶음)과 접근(이름으로 값을 꺼냄).

var beverage = "espresso";   // 선언 + 초기화
print beverage;                // 접근 → espresso
var empty;                     // 초기화 없으면 nil

선언은 아무 데나 올 수 없다 — declaration 계층 추가

program→declaration* EOF
declaration→varDecl | statement NEW
varDecl→"var" IDENTIFIER ( "=" expression )? ";" NEW
statement→exprStmt | printStmt
primary→... | IDENTIFIER NEW
왜 declaration을 따로?  if (cond) var x = 1; 처럼 변수 선언이 단독으로 숨어드는 걸 막기 위해서입니다. declaration → varDecl | statement 로 한 겹을 두면, "변수 선언이 허용되는 자리"와 "일반 문장만 되는 자리"가 문법적으로 구분됩니다.

3Environment — 변수를 담는 그릇

변수 이름 → 값의 묶음(binding)을 저장하는 자료구조. 본질은 HashMap<String, Object> 하나입니다.

class Environment {
  final Environment enclosing;              // 바깥(부모) 스코프
  private final Map<String,Object> values = new HashMap<>();

  void define(String name, Object value) { ... }  // 새 변수 묶기 (재정의 허용)
  Object get(Token name)             { ... }  // 값 꺼내기
  void assign(Token name, Object v)  { ... }  // 기존 변수 수정
}

define — 선언

맵에 그냥 put. 이미 있으면 덮어씀. 그래서 전역에서 var a = 1; var a = 2; 가 허용됨.

get / assign — 접근/수정

이름이 없으면 RuntimeError("Undefined variable"). 단, 컴파일 에러가 아닌 런타임 에러로 처리해 재귀 같은 패턴을 가능하게 함.

변수는 토큰이 아니라 lexeme(문자열)로 저장됩니다. 코드 곳곳의 a 토큰은 서로 다른 객체지만, 같은 문자열 "a"로 묶이므로 같은 바인딩을 가리킵니다.

4할당 (Assignment)

Lox에서 할당은 문장이 아니라 표현식입니다. 값을 돌려주므로 a = b = 3 처럼 연쇄 가능(우결합).

expression→assignment NEW
assignment→IDENTIFIER "=" assignment | equality NEW

파서의 트릭 — l-value 문제

= 왼쪽이 단순 변수가 아니라 a.b.c = 3 처럼 복잡할 수 있어, 미리 "할당이다"라고 알기 어렵습니다. 그래서 책의 방법은:

1

왼쪽을 그냥 표현식으로 파싱

일단 equality까지 일반 표현식으로 읽는다

2

= 가 보이면?

방금 읽은 게 할당 대상이었음을 깨닫는다

3

대상이 Variable인지 검사

맞으면 Expr.Assign 으로 변환, 아니면 에러

declare vs assign  define은 없으면 만든다. assign은 없으면 에러다. 즉 할당은 새 변수를 암묵적으로 만들지 않는다 — 오타로 인한 변수 생성을 방지.

5스코프 & 블록 — 환경의 사슬

블록 { } 마다 새 Environment를 만들고, 그것이 바깥 환경(enclosing)을 가리키게 합니다. 이렇게 만들어지는 사슬이 렉시컬 스코프를 구현합니다. 

var a = "global";
{
  var a = "block";   // 바깥 a를 가림 (shadowing)
  print a;            // block
}
print a;              // global  (블록 끝나면 복원)

실행 중 환경 사슬의 모습

Block Env현재(innermost)
a"block"
enclosing ──┐
→enclosing
Global Env최상위
a"global"
enclosing = null

탐색은 안쪽 → 바깥쪽

get/assign은 현재 환경부터 찾고, 없으면 enclosing으로 재귀적으로 거슬러 올라감. 안쪽이 먼저 매칭되므로 자연스럽게 shadowing이 구현됨.

블록이 끝나면 복원

executeBlock()은 새 환경으로 바꿔 실행하고, finally에서 이전 환경으로 되돌림. 에러가 나도 환경이 새지 않음.

이 구조는 한 줄로 보면 사슬이지만, 전체 실행에 걸쳐 보면 여러 갈래로 뻗는 "선인장 스택(cactus stack)" 트리를 이룹니다.

추가된 블록 문법

statement→exprStmt | printStmt | block NEW
block→"{" declaration* "}" NEW
