1왜 9장인가?

8장 언어는 위→아래로 한 번 흐를 뿐. 건너뛰거나 반복할 수 없다.

var a = 1;
print a;       // 항상 순서대로 한 번씩만 실행
print a + 1;   // 조건 분기도, 반복도 불가능
튜링 완전의 조건 = 산술 + 제어 흐름 + (이론상) 무한한 메모리. 9장이 "제어 흐름"을 채우면 Lox는 계산 가능한 모든 함수를 표현할 수 있게 된다.

3조건 실행 — if / else

조건이 참이면 then, 거짓이면 else. else는 선택사항.

if (condition) {
  print "참일 때";
} else {
  print "거짓일 때";
}
statement→exprStmt | ifStmt | printStmt | block NEW
ifStmt→"if" "(" expression ")" statement
( "else" statement )? NEW

4⚠️ 단글링 else 문제

if (first) if (second) A(); else B(); — else는 어느 if에 붙나?

해석 A — 안쪽 if ✅
if (first)
  if (second) A();
  else B();
해석 B — 바깥 if ❌
if (first)
  if (second) A();
else B();
규칙  else는 가장 가까운 앞쪽 if에 묶인다. 파서가 else를 보자마자 즉시(eagerly) 현재 if에 매칭하면 자동 해결.

5논리 연산자 — and / or

일반 이항 연산자와 다르다: 단락 평가 + 불린이 아닌 피연산자 값 반환

assignment→... | logic_or
logic_or→logic_and ( "or" logic_and )* NEW
logic_and→equality ( "and" equality )* NEW

값을 돌려준다

print "hi" or 2;   // "hi"
print nil or "y";  // "y"

true가 아니라 피연산자를 반환.

우선순위

or 가 and 보다 낮다.
assignment → or → and → equality → ...

왜 Expr.Binary 재사용 안 함? 일반 연산자는 양쪽을 먼저 평가 → 논리 연산자는 왼쪽 먼저 보고 필요할 때만 오른쪽 평가. 평가 방식이 달라 Expr.Logical 별도 필요.

7while 루프

조건이 참인 동안 본문을 반복 — 가장 단순한 반복

var i = 0;
while (i < 3) {
  print i;
  i = i + 1;
}
statement→... | whileStmt | ... NEW
whileStmt→"while" "(" expression ")" statement NEW
visitWhileStmt는 Java의 while로 감싼다 → 소스 길이와 무관하게 실행 시간이 늘어날 수 있게 됨.

8for = 문법 설탕 (디슈가링)

for는 새 AST 노드를 안 만든다. 기존 while + block으로 변환될 뿐.

forStmt→"for" "(" ( varDecl | exprStmt | ";" )
expression? ";" expression? ")" statement NEW

세 절:  초기화  ;  조건  ;  증감  — 모두 생략 가능

디슈가링의 이점  프론트엔드(파서)에서 변환만 하면 백엔드(인터프리터)는 손댈 필요 없음. 편의 문법을 공짜로.

9디슈가링 변환 (절을 클릭/호버)

원본 for문
for (var i = 0; i < 10; i = i+1)
  print i;
➜
디슈가링 결과
{
  var i = 0;
  while (i < 10) {
    print i;
    i = i+1;
  }
}
초기화 → 바깥 블록 맨 앞 조건 → while 조건 증감 → 본문 끝 본문 → while 본문 앞

10추가된 AST 노드

단 3개. for는 디슈가링되므로 전용 노드 없음.

Stmt.If

조건 분기. elseBranch nullable.

Expr condition, Stmt thenBranch, Stmt elseBranch

Stmt.While

조건 참인 동안 body 반복.

Expr condition, Stmt body

Expr.Logical

and/or. 단락 평가 위해 Binary와 분리.

Expr left, Token operator, Expr right

Stmt.For ❌

없음! 디슈가링으로 while+block에 흡수.

정리

핵심 4가지

  • 제어 흐름(분기+반복)이 추가되며 Lox가 튜링 완전해짐
  • 단글링 else는 "가장 가까운 if"에 묶어 해결
  • and/or는 단락 평가 + 피연산자 값 반환 → Expr.Logical 별도
  • for는 문법 설탕 — while+block으로 디슈가링, 전용 노드 없음
