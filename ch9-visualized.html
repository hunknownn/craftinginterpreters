Chapter 9 — Control Flow

분기와 반복으로 인터프리터에 튜링 완전성을 부여하다

if/else · and/or 단락평가 · while · for 디슈가링

⓪ 튜링 완전 ① if / else ② 논리 연산자 ③ while ④ for 디슈가링 🔌 단락평가 데모 🌳 AST 노드

8장까지의 언어는 변수를 기억하고 출력할 수 있지만, 코드는 위에서 아래로 한 번 흐를 뿐입니다. 조건에 따라 건너뛰거나, 같은 코드를 반복할 수 없죠. 9장은 제어 흐름(control flow) 을 추가해 언어를 비로소 튜링 완전하게 만듭니다.

튜링 완전의 조건  산술 + 약간의 제어 흐름 + (이론상) 무한한 메모리. 9장에서 "제어 흐름"이 채워지면 Lox는 계산 가능한 모든 함수를 표현할 수 있게 됩니다.

제어 흐름의 두 갈래

분기 (Branching)

조건에 따라 코드를 실행하거나 건너뛴다. → if, and/or

반복 (Looping)

코드를 여러 번 실행한다. → while, for

1조건 실행 — if / else

조건이 참이면 then 가지를, 거짓이면 else 가지를 실행. else는 선택사항입니다.

if (condition) {
  print "참일 때";
} else {
  print "거짓일 때";
}
statement→exprStmt | ifStmt | printStmt | block NEW
ifStmt→"if" "(" expression ")" statement ( "else" statement )? NEW

⚠️ 단글링 else 문제 (dangling else)

아래 코드에서 else는 어느 if에 붙을까요? 문법만으로는 모호(ambiguous) 합니다.

if (first) if (second) whenTrue(); else whenFalse();
해석 A — 안쪽 if에 붙음
if (first)
  if (second) whenTrue();
  else whenFalse(); ✅
해석 B — 바깥 if에 붙음
if (first)
  if (second) whenTrue();
else whenFalse(); ❌
해결  "else는 가장 가까운 앞쪽 if에 묶인다." 파서가 else를 보자마자 즉시(eagerly) 현재 if에 매칭하면 자연스럽게 이 규칙이 적용됩니다 — 별도 처리 불필요.

2논리 연산자 — and / or

and, or는 일반 이항 연산자와 다릅니다. 단락 평가를 하고, 불린이 아니라 피연산자 값을 돌려주기 때문이죠.

assignment→IDENTIFIER "=" assignment | logic_or
logic_or→logic_and ( "or" logic_and )* NEW
logic_and→equality ( "and" equality )* NEW

우선순위: or 가 and 보다 낮다 (assignment → or → and → equality → ...).

① 단락 평가 (short-circuit)

왼쪽만 보고 결과가 정해지면 오른쪽은 평가조차 안 함.

// or: 왼쪽이 truthy면 끝
true or sideEffect();  // 호출 안 됨
// and: 왼쪽이 falsey면 끝
false and sideEffect(); // 호출 안 됨

② 값을 돌려준다 (불린 아님)

"적절한 truthiness를 가진 피연산자"를 반환.

print "hi" or 2;   // "hi" (true 아님!)
print nil or "yes"; // "yes"
print nil and 2;   // nil
왜 Expr.Binary를 재사용 안 하나?  일반 이항 연산자는 양쪽을 먼저 평가한 뒤 계산합니다. 하지만 논리 연산자는 왼쪽을 먼저 보고, 필요할 때만 오른쪽을 평가해야 하므로 평가 방식 자체가 달라 별도의 Expr.Logical 노드가 필요합니다.

3while 루프

조건이 참인 동안 본문을 반복. 가장 단순한 반복 형태입니다.

var i = 0;
while (i < 3) {
  print i;
  i = i + 1;
}
statement→... | whileStmt | ... NEW
whileStmt→"while" "(" expression ")" statement NEW
인터프리터의 visitWhileStmt는 Java의 while 루프로 감쌉니다. 소스 코드 길이와 무관하게 실행 시간이 늘어날 수 있게 되어 — 이 지점에서 진짜 "반복 계산"이 가능해집니다.

4for 루프 = 문법 설탕 (디슈가링)

for문은 새 AST 노드를 만들지 않습니다. 이미 있는 while·block으로 변환(desugar) 될 뿐이죠. 인터프리터는 for를 전혀 몰라도 됩니다.

forStmt→"for" "(" ( varDecl | exprStmt | ";" )
expression? ";" expression? ")" statement NEW

세 절: 초기화 ; 조건 ; 증감 — 모두 생략 가능.

디슈가링 변환 (절을 클릭해 대응 관계 확인)

원본 for문
for (var i = 0; i < 10; i = i + 1)
  print i;
➜
디슈가링된 결과
{
  var i = 0;
  while (i < 10) {
    print i;
    i = i + 1;
  }
}

파서가 거꾸로 조립하는 순서

1

본문 + 증감

증감이 있으면 본문을 { body; 증감; } 블록으로 감싼다

2

조건 → while

조건으로 while 생성. 조건 생략 시 true(무한루프)

3

초기화로 감싸기

초기화가 있으면 전체를 { 초기화; while... } 블록으로

디슈가링의 이점  프론트엔드(파서)에서 기존 노드로 변환만 하면, 백엔드(인터프리터)는 건드릴 필요가 없습니다. 편의 문법을 공짜로 얻는 셈 — Lisp, C++, Ruby 등이 정도의 차이는 있어도 같은 전략을 씁니다.

🔌 단락 평가 인터랙티브 데모

버튼을 눌러, and/or가 오른쪽을 평가하는지 건너뛰는지 단계별로 확인하세요. sideEffect()는 호출되면 로그를 남깁니다.

▶ 버튼을 눌러 평가를 시작하세요.

🌳 9장에서 추가된 AST 노드

단 3개. for문은 디슈가링되므로 전용 노드가 없습니다.

Stmt.If

조건 분기. elseBranch는 nullable.

Expr condition, Stmt thenBranch, Stmt elseBranch

Stmt.While

조건이 참인 동안 body 반복.

Expr condition, Stmt body

Expr.Logical

and/or. 단락 평가 위해 Binary와 분리.

Expr left, Token operator, Expr right

Stmt.For ❌
